# План переработки: индивидуальные ставки специалистов

> Живой документ. Заполняется по ходу обсуждения: первая часть —
> постановка задачи и варианты, вторая — принятые решения, третья —
> разбиение на этапы и чек‑лист реализации.

---

## 1. Цель и контекст

### Бизнес‑проблема

Сейчас стоимость роли в услуге определяется грейдом (`grades_table.UF_RATE`):
все специалисты одного грейда оцениваются одинаково. В реальности
**ставка у каждого специалиста своя**, поэтому текущий расчёт
получается «средняя по больнице» и для принятия решения по проекту
малоприменим.

### Что хотим получить

1. Каталог услуг перестаёт быть источником стоимости — там остаётся
   только состав работ (роли + нормативные часы).
2. Стоимость возникает **в корзине**, после того как пользователь
   ввёл ставки конкретных специалистов.
3. Ставки сохраняются вместе с черновиком (т.е. являются частью
   проекта‑оценки, а не глобальным справочником).
4. Поддерживаются сценарии вида «одну и ту же услугу делают двое
   специалистов с разными ставками и разным распределением часов».
5. Поведение «общих» и «публичных» черновиков не должно ломаться:
   ставки конкретных людей — потенциально чувствительные данные.

## 2. Как устроено сейчас (отправная точка)

```
catalog → services (iblock)
        → sostav_table (UF_SERVICE, UF_ROLE, UF_HOURS, UF_RESULT)
        → roles_table (UF_NAME, UF_DEPARTMENT)
        → grades_table (UF_NAME, UF_RATE)           ← источник цены
        → role_default_grades (UF_ROLE, UF_GRADE)

корзина (сессия) для одной услуги:
{
  NAME, SERVICE_LEVEL, ROOT_SECTION_CODE, SECTION_NAME,
  ROLES: {
    <roleId>: {
      ROLE_ID, ROLE_NAME, RESULT,
      HOURS, STD_HOURS,
      GRADE_ID, RATE, COST            ← всё расчётное
    }
  }
}
```

Текущая стоимость роли = `RATE × HOURS`, стоимость услуги =
сумма по ролям × коэффициент уровня (0.77/1.0/1.3).

## 3. Целевая модель (на уровне понятий)

Появляются три новых понятия:

- **Команда проекта** — список специалистов, собранный для данного
  черновика. Существует в рамках одной корзины/черновика. Управляется
  через модалку с битриксовым пикером пользователей.
- **Специалист проекта** — запись в команде. Содержит:
  - привязку к Bitrix‑пользователю (имя, аватар) и/или ad‑hoc‑имя
    (см. развилку G в разделе 4),
  - **индивидуальную ставку (₽/час)**,
  - **необязательную категорию (грейд)** — как информационный ярлык
    (отображается в карточках и Excel, в расчёте не участвует).
- **Назначение** (assignment) — «специалист X делает роль Y в услуге Z
  N часов». Это единица, по которой считается стоимость.

```
Услуга в корзине
└─ Роль 1 (Дизайнер, std 40ч)
   ├─ Назначение A → Иванов (Senior), 24ч × 5 500 ₽
   └─ Назначение B → Петров (Middle), 16ч × 8 000 ₽
└─ Роль 2 (Аналитик, std 20ч)
   └─ Назначение C → Сидорова, 20ч × 6 000 ₽
```

«Два специалиста на одну роль» = просто два элемента в массиве
`assignments` этой роли. UI каталога и корзины при разворачивании
карточки услуги показывает один и тот же контент — состав ролей
с назначениями (см. раздел [UX‑поток](#7-ux-поток)).

## 4. Развилки (варианты по осям)

Каждая ось рассматривается независимо; финальная архитектура —
комбинация выбранных вариантов.

### A. Где хранится список специалистов

| # | Вариант | Плюсы | Минусы |
|---|---|---|---|
| **A1** | Per‑line. У каждой строки корзины свой набор специалистов. | Локально, проще модель. | Один и тот же специалист задаётся N раз по числу услуг → дублирующиеся ставки, расхождения. |
| **A2** ✅ | Project‑roster. У черновика отдельный список «команда проекта»; в строках выбирается, кто из команды. | Ставка вводится один раз и переиспользуется. Видна общая структура проекта. | Дополнительная UI‑секция, лишний шаг при «быстро посчитать одну услугу». |
| **A3** | Гибрид. Можно ввести ad‑hoc прямо в назначении, потом «вытащить» в команду проекта. | Гибко. | Дороже UI, выше когнитивная нагрузка. |

**Решение:** A2. Команда собирается в модалке с битриксовым выбором
пользователей, ставка указывается там же.

### B. Как обыгрываем «двух специалистов на одной роли в одной услуге»

| # | Вариант | Плюсы | Минусы |
|---|---|---|---|
| **B1** | Добавить услугу дважды. | Не трогаем схему. | UX‑шум: два карточки одной услуги, неудобно в отчётах, путаница. |
| **B2** ✅ | Массив назначений на роль. UI: внутри роли — кнопка «+ ещё специалист». | Чисто моделирует реальность. Excel‑экспорт выглядит органично (одна задача — несколько строк). | Большой UI‑рефакторинг карточек корзины. |
| **B3** | Часы роли делятся между N специалистами (общая сумма часов фиксирована). | Сохраняет «бюджет часов из каталога». | Жёстко: иногда нужно, наоборот, увеличить общие часы при втором специалисте. |

**Решение:** B2. К каждой роли в услуге можно выбрать одного или
нескольких специалистов из команды; каждому ставятся свои часы.

### C. Что делаем с грейдами (категории персонала)

| # | Вариант | Плюсы | Минусы |
|---|---|---|---|
| **C1** | Удаляем сущность. | Меньше путаницы. | Ломаем существующие данные в HL и привязки `role_default_grades`. |
| **C2** ✅ | Оставляем как «ярлык/категорию» без ставки. Грейд указывается у **специалиста** (в карточке команды), не у роли. UF_RATE из грейда не используется. | Грейд переезжает на нужный уровень (характеристика человека), остаётся в Excel. | Связь `role_default_grades` теряет смысл — можно почистить позже. |
| **C3** | Используем `UF_RATE` как **дефолтную подсказку** при создании специалиста. | Удобно «быстро добавить дизайнера с предсказуемой ставкой». | Возвращает «среднюю по больнице» в UX, размывает идею. |

**Решение:** C2 с уточнением — грейд указывается у специалиста при
добавлении в команду, как опциональное поле. Используется как метка
в карточках и колонка в Excel. На расчёт не влияет.

### D. Что показывает каталог без цен

| # | Вариант | Плюсы | Минусы |
|---|---|---|---|
| **D1** | Только нормативные часы по ролям. Никаких цен/ставок. | Чисто и однозначно. | Лишает пользователя интуиции «дорого/дёшево» на этапе выбора. |
| **D2** | Часы + бейдж «Стоимость рассчитывается в корзине». | То же + явный сигнал. | UI чуть шумнее. |
| **D3** | Часы + опциональная «справочная» цифра из старых грейдов. | Сохраняет ориентир. | Возвращает источник «средней цены», который мы хотим убрать. |
| **D4** ✅ | **Контекстный каталог**: услуга в каталоге имеет два режима отображения. До добавления в корзину — состав ролей и нормативные часы, без сумм. После добавления (или при наличии команды) — разворачивается с фактическими назначениями и стоимостью. | Один UI охватывает «справочник» и «настройку проекта». | Сложнее реализация: каталог должен знать состояние корзины и команды. |

**Решение:** D4 — два режима карточки услуги в каталоге («не настроена»
и «настроена»). При разворачивании настроенной услуги показываем
назначения и стоимость, идентично карточке в корзине.

### E. Что с карточкой «добавить в корзину» в каталоге

| # | Вариант | Плюсы | Минусы |
|---|---|---|---|
| **E1** | Один клик «В корзину». Все правки — в корзине. | Минимум форм, проще каталог. | Если у пользователя нет привычки, может «потерять» услугу в корзине. |
| **E2** | В каталоге настраиваются только часы и уровень обслуживания (без ставок). | Знакомый сценарий с минимумом изменений. | Удваивает UI: часы можно менять и в каталоге, и в корзине. |
| **E3** ✅ | После клика «В корзину» карточка раскрывается прямо в каталоге и показывает таблицу назначений, идентичную корзинной. | Единый UI везде, не нужно прыгать между страницами. | Каталог становится «жирнее» (полный редактор). |

**Решение:** E3 в связке с D4. Кнопка «В корзину» (1 клик) меняет
карточку с режима «не настроена» на «настроена + раскрыта», где
сразу можно назначать специалистов.

### F. Уровень обслуживания (low/medium/high)

| # | Вариант | Плюсы | Минусы |
|---|---|---|---|
| **F1** ✅ | Оставляем; коэффициент применяется к сумме всех назначений услуги. | Минимум изменений. | Логика прежняя, понятная. |
| **F2** | Коэффициент применяется к каждому назначению. | Эквивалентно F1 (математически), но UI явно показывает каждую строку с коэффициентом. | Лишний шум в строках. |
| **F3** | Убираем уровень совсем — пусть ставки сами всё отражают. | Чище. | Теряем «уровень обслуживания» как метрику; шаблоны в Excel меняются. |

**Решение по умолчанию:** F1. Можно пересмотреть после MVP.

### G. Специалист — Bitrix‑пользователь или произвольная строка

| # | Вариант | Плюсы | Минусы |
|---|---|---|---|
| **G1** | Только текст (Имя). | Простота. | Нет аватара/связи с порталом, опечатки. |
| **G2** ✅ | Только Bitrix‑пользователи. | Аватары, единое имя на портале. | Нельзя посчитать стоимость с участием подрядчика, фрилансера и т.п. |
| **G3** | Оба: или выбор из портала, или ad‑hoc‑имя. | Покрываются все кейсы. | Сложнее UI/валидация. |

**Решение:** G2. Только сотрудники портала. Подрядчиков/фрилансеров —
не поддерживаем в MVP (если потребуется, добавим G3 позже).

### H. Поведение при пустых ставках

| # | Вариант | Плюсы | Минусы |
|---|---|---|---|
| **H1** ✅ | Считаем сумму как 0 в строке. Итог — частичный. | Гибко: можно сохранить «скелет проекта», добить ставки позже. | Можно случайно отправить заказчику недосчитанный расчёт. |
| **H2** | Сумма не отображается, пока не введены все ставки. | Защита от ошибок. | Раздражает на этапе планирования. |
| **H3** | Запрет на сохранение/экспорт черновика с пустыми ставками. Жёсткий ворота. | Качество данных. | Нельзя сделать «черновик‑заготовку». |
| **H4** | H1 + предупреждающий баннер + блокировка Excel. | Баланс. | UI‑оверхед. |

**Решение:** H1 — частичный итог без баннеров/блокировок. Простота
важнее «защиты от случайной отправки».

### I. Миграция старых черновиков

| # | Вариант | Плюсы | Минусы |
|---|---|---|---|
| **I1** | Read‑only режим для старого формата. | Чёткая граница. | Усложняет код. |
| **I2** | Лазивная конвертация: грейд → «виртуальный» специалист со ставкой из `UF_RATE`. | Преемственность. | «Фейковые» специалисты, путаница. |
| **I3** ✅ | Удалить старые черновики. | Чистый старт, простой код. | Потеря пользовательских данных. |

**Решение:** I3. Старые черновики стираются при деплое. Код
`DraftService` сразу пишется под формат v2 без поддержки v1.

> **Внимание:** перед выкаткой нужно очистить таблицу
> `drafts_table` (TRUNCATE через админку Bitrix или скрипт).
> Этот шаг отдельно отмечен в этапах реализации (см. ниже).

### J. Защита ставок в shared/public черновиках

Ставки могут содержать имена и зарплаты — это чувствительно. Возможные политики:

| # | Вариант | Плюсы | Минусы |
|---|---|---|---|
| **J1** ✅ | Никаких ограничений: shared/public видят всё. | Просто. | Утечка ставок. |
| **J2** | В public — скрывать имена специалистов. | Скрытие чувствительного. | Усложняет логику отображения. |
| **J3** | В public — скрывать имена и ставки. | Максимум защиты. | Бесполезно для оценки. |
| **J4** | Запретить тип `public` для черновиков со ставками. | Простая политика. | Лишает функционала. |

**Решение по умолчанию:** J1. Можно пересмотреть после обратной
связи от пользователей.

### K. Удаление специалиста, у которого есть назначения

| # | Вариант | Плюсы | Минусы |
|---|---|---|---|
| **K1** | Запретить удаление. Сообщение «специалист задействован в N услугах». | Безопасно. | Требует дополнительных действий от пользователя. |
| **K2** ✅ | Каскадно удалить все назначения этого специалиста. Услуги остаются; роли, где он был единственным, становятся пустыми; роли, где он был одним из, остаются с другими специалистами. | Естественно для пользователя: «убрал из команды — больше нигде не учитывается». | Молчаливое изменение часов в услугах. |
| **K3** | Сохранить назначения, но `specialistId=null` (orphan). | Не теряем часы. | Слот без специалиста — в стоимости не участвует, выглядит так же, как K2 (роль пустеет). |

**Решение:** K2. При удалении специалиста все его назначения
удаляются. Если роль остаётся без назначений — роль показывается
пустой (часы по нормативу видны, в стоимость не входит). Услуга
остаётся в корзине. Перед удалением — подтверждающий диалог, если
есть назначения.

### L. UX‑поток первого касания (guided vs free)

| # | Вариант | Плюсы | Минусы |
|---|---|---|---|
| **L1** | Жёсткий guided: нельзя добавлять услуги, пока не создана команда. | Гарантирует наличие данных. | Раздражает «справочное» использование каталога. |
| **L2** ✅ | Свободный поток: команду и услуги можно собирать в любом порядке и параллельно. Подсказка в UI (баннер «Соберите команду, чтобы увидеть стоимость»). | Гибкость. | Пользователь может «забыть» собрать команду. |
| **L3** | Мягкий guided: первый заход на каталог показывает приветственный экран с предложением собрать команду; можно пропустить. | Баланс. | Дополнительный экран. |

**Решение по умолчанию:** L2. Поток свободный, с подсказкой‑баннером
в каталоге, пока команда пуста.

## 5. Целевая структура данных

### 5.1. Корзина / черновик (новая)

```jsonc
{
  "version": 2,
  "team": [
    {
      "id": "sp_1",                  // локальный uid внутри корзины
      "bitrixUserId": 42,            // обязательно (G2)
      "rate": 5500,                  // ₽/час
      "gradeId": 7                   // null если не указан
    },
    {
      "id": "sp_2",
      "bitrixUserId": 73,
      "rate": 8000,
      "gradeId": null
    }
  ],
  "services": {
    "123": {                          // serviceId из каталога
      "name": "Дизайн UI",
      "rootSectionCode": "agile",
      "sectionName": "Проектирование",
      "serviceLevel": "medium",
      "roles": {
        "7": {                        // roleId
          "roleId": 7,
          "roleName": "Дизайнер",
          "stdHours": 40,             // оригинал из sostav_table — справочно
          "assignments": [
            { "id": "as_1", "specialistId": "sp_1", "hours": 24 },
            { "id": "as_2", "specialistId": "sp_2", "hours": 16 }
          ]
        },
        "11": {
          "roleId": 11,
          "roleName": "Аналитик",
          "stdHours": 20,
          "assignments": []           // пусто — например, после удаления спеца
        }
      }
    }
  }
}
```

Заметки:
- `team[].bitrixUserId` — единственный идентификатор человека (G2).
  Имя/аватар/email не дублируем в JSON — подтягиваем при отображении
  из `CUser::GetByID()` и аналогов. Это уменьшает риск рассинхрона
  и устаревших данных.
- `team[].gradeId` — `null` или ID из `grades_table`. Используется
  только для отображения метки в карточках и колонке Excel
  (C2 — на расчёт не влияет).
- `assignments` всегда массив, может быть **пустым** после удаления
  специалиста (K2). UI в этом случае показывает строки роли как
  «нет назначений», предлагает «+ выбрать специалиста».
- При добавлении новой услуги в корзину `assignments` инициализируется
  одним элементом `{ specialistId: null, hours: stdHours }`.
- В назначении нет своего `rate` — он берётся из специалиста.
  Изменение ставки в команде пересчитывает все услуги.
- Если `specialistId = null`, назначение «не оценено»: в стоимость
  не входит (H1), но часы видны.
- Один и тот же `bitrixUserId` не может встречаться в команде дважды
  (валидация на бэке и UI).
- Один специалист не может появляться на одной роли в одной услуге
  дважды (валидация на бэке и UI).

### 5.2. Расчёт стоимости

```text
assignmentCost(a, specialists) = specialists[a.specialistId].rate × a.hours
                                  (если specialistId есть и rate>0; иначе 0)

roleCost(role, specialists)    = Σ assignmentCost(a)

serviceBaseCost(service, sps)  = Σ roleCost(role) for role in service.roles

serviceCost(service, sps)      = serviceBaseCost × levelCoeff(serviceLevel)

cartTotal(cart)                = Σ serviceCost(service)
```

### 5.3. Что хранится в `drafts_table.UF_DATA`

Прежний формат — словарь `serviceId → cartItem` без обёртки. Новый
формат — объект `{ version: 2, team, services }`. По решению I3
старые черновики стираются при деплое, поэтому конвертер из v1 в
v2 **не пишется** — `DraftService` читает только формат с
`version: 2`, остальные считает несуществующими (HTTP 404).

## 6. Влияние на компоненты

### `Repository`
- В `getServiceComposition()` не подмешивать `RATE`/`COST` — стоимости
  в каталоге больше нет, возвращаем только `{roleId, roleName, hours, result}`.
- `getGrades()` оставить как есть, но `UF_RATE` не используем
  (C2 — грейд только как метка).
- `role_default_grades` — больше не используется. Не удаляем
  таблицу/привязки, но игнорируем в коде.

### `CostCalculator`
- Полная переработка. Новый API:
  ```php
  assignmentCost(array $assignment, array $team): float
  roleCost(array $role, array $team): float
  serviceCost(array $service, array $team): float
  cartTotal(array $cart): float
  ```
- `$team` — `[ specialistId => ['rate' => 5500, 'gradeId' => 7, ...] ]`.
- Старые методы удаляем (по I3 черновиков со старым форматом нет).

### `CartService`
- Хранение в сессии: `$_SESSION['SERVICE_CART_<id>'] = ['team' => [...], 'services' => [...]]`.
- Новые методы:
  - **Команда:** `getTeam()`, `addTeamMember($bitrixUserId, $rate, $gradeId = null)`,
    `updateTeamMember($specialistId, $data)`, `removeTeamMember($specialistId)`.
  - **Назначения:** `addAssignment($serviceId, $roleId, $specialistId = null, $hours = null)`,
    `updateAssignment($serviceId, $roleId, $assignmentId, $data)`,
    `removeAssignment($serviceId, $roleId, $assignmentId)`.
  - **Услуги:** `add($serviceId)` — создаёт одно `{specialistId: null, hours: stdHours}`
    на каждую роль. `remove($serviceId)`.
- `removeTeamMember()` — каскадно удаляет назначения этого специалиста
  во всех услугах (K2). Возвращает количество затронутых услуг для
  подтверждающего диалога.
- `replace($data)` — поддержка только формата v2.

### `DraftService`
- `create`/`updateData` — принимают новый JSON v2.
- `load()` — если `UF_DATA.version !== 2` → возвращает «not found».
- При деплое запускаем `TRUNCATE drafts_table` (см. этап 6 ниже).

### `ExcelExporter`
- Шапка таблицы: «Раздел / Услуга / Роль / Специалист / Грейд /
  Часы / Ставка / Стоимость».
- Одна строка на назначение. Слияние ячеек на уровне услуги/роли
  (для читаемости).
- Если `specialistId = null` — в колонке «Специалист» ставим `—`,
  ставка/стоимость пустые.
- Грейд берётся из специалиста (`team.gradeId`).

### `service.catalog` (component + template + JS)
- Карточки услуг без цен (D4 «не настроена»).
- Кнопка «В корзину» — один клик; после добавления карточка
  переключается в режим «настроена» (E3) и показывает редактор
  назначений в каталоге.
- В шапке — индикатор «Команда проекта» с числом.
- Подсказка‑баннер «Соберите команду, чтобы увидеть стоимость»,
  пока команда пуста (L2).

### `service.cart` (component + template + JS)
- Список услуг сгруппирован по корневому разделу (как сейчас).
- Каждая карточка услуги — тот же редактор, что и в каталоге (общий
  JS‑компонент).
- В шапке — кнопка «Команда проекта», открывает модалку.
- Блок «Итого» — суммирует ассайнменты с `specialistId != null`.
- Excel/черновики работают без ограничений по пустым ставкам (H1).

### Модалка команды проекта (новый компонент)
- Список текущих специалистов: аватар, имя, грейд (опц.), ставка inline.
- Кнопка «+ Добавить специалиста» → стандартный битриксовый
  user‑picker (BX UI; ищем готовый — `BX.UI.UserSelectorPopup` либо
  через `bitrix:main.ui.selector`).
- Поле ставки — обязательное при добавлении.
- Поле грейд — необязательное, dropdown из `grades_table`.
- Кнопка «Удалить» с подтверждением, если есть назначения
  (показываем сколько и в каких услугах).
- Сохранение — AJAX в `service.cart` (модалка переиспользуется на
  обеих страницах).

## 7. UX‑поток

### Где живёт «Команда проекта»

Кнопка/индикатор «Команда проекта» располагается **в шапке и
каталога, и корзины** — пользователь видит её всегда, может открыть
в любой момент. Бейдж показывает число специалистов.

```
[ Каталог услуг ЦК ПР ]                              [👥 Команда: 0]
```

Клик открывает **модалку команды**:

1. Список добавленных специалистов: аватар, имя, грейд (опц.), ставка
   (редактируется inline), кнопка «Удалить».
2. Кнопка «+ Добавить специалиста».
3. При добавлении — открывается стандартный пикер пользователей Bitrix
   (поиск по сотрудникам портала). После выбора — поле «Ставка ₽/час»
   и выпадающий список «Грейд» (необязательный).
4. Кнопка «Сохранить» / «Закрыть».

### Карточка услуги: два режима

В каталоге и в корзине одна и та же услуга рендерится одной
JS‑компонентой с двумя визуальными состояниями.

**Режим «не настроена»** (услуга не в корзине):
```
🧩 Дизайн UI                                  [В корзину]
   • Дизайнер: 40 ч (норматив)
   • Аналитик: 20 ч (норматив)
```

**Режим «настроена»** (услуга в корзине):
```
🧩 Дизайн UI                          Σ 308 800 ₽   [Убрать]
  ▼ Подробнее
   Дизайнер · норматив 40 ч
      [Иванов ▼]   [24] ч × 5500 = 132 000 ₽   ✕
      [Петров ▼]   [16] ч × 8000 = 128 000 ₽   ✕
      [+ ещё специалист]
   Аналитик · норматив 20 ч
      [Сидорова ▼] [20] ч × 6000 = 120 000 ₽   ✕
      [+ ещё специалист]
```

В корзине (`/catalog/basket/`) разворачивание карточки даёт тот же
блок. На страницах различия только в обвязке: список услуг в корзине
сгруппирован по корневому разделу + блок «Итого» + Excel/черновики.

### Поведение «команда → услуги» (не жёсткий guided)

Пользователь может:

- **Сначала собрать команду**, потом добавлять услуги (рекомендуемый
  сценарий: при добавлении услуги назначения автоматически предлагают
  выбор из команды).
- **Сначала добавить услуги**, потом собирать команду (тогда
  назначения остаются «не оценены», по ходу заполнения команды можно
  привязывать).
- **Параллельно** — модалка команды доступна всегда.

Если пользователь пытается выбрать специалиста, которого ещё нет в
команде, в выпадающем списке последним пунктом будет «+ Добавить в
команду…» — он откроет модалку команды поверх текущей карточки.

### Связь «специалист удалён → его назначения» (K2)

При удалении специалиста из команды все его назначения каскадно
удаляются:

- Если у роли остаются другие назначения — роль продолжает работать
  как обычно (показывает оставшихся специалистов).
- Если назначение было единственным — массив `assignments` становится
  пустым. Роль показывается как «нет назначений», часы по нормативу
  видны, в стоимость не входит. Пользователь может нажать «+ выбрать
  специалиста» и заполнить.
- Услуга **не удаляется** из корзины.

Перед удалением — диалог подтверждения: «Иванов И.И. участвует в 3
услугах (5 назначений). Удалить из команды? Все его назначения будут
сняты.»

## 8. Этапы реализации

Каждый этап — самостоятельный коммит (или серия близких). После
этапа репозиторий компилируется; функционал может быть «полусломан»
до этапа 5, но это допустимо в рамках одной фича‑ветки.

### Этап 1 — серверный слой (модель + расчёт)

Файлы:
- `local/php_interface/include/lib/Catalog/CostCalculator.php` —
  переписать под `team`+`assignments`. Удалить старые методы.
- `local/php_interface/include/lib/Catalog/CartService.php` —
  переписать структуру сессии, добавить CRUD команды и назначений.

  убрать `RATE`/`COST` из `getServiceComposition()`.
- `local/php_interface/include/lib/Catalog/DraftService.php` —
  только формат v2, отказ от старого.

Критерий готовности: проект собирается, юнит‑логика расчёта может
быть проверена «на бумаге» (без UI).

### Этап 2 — AJAX‑контракты (component `service.cart`)

  - оставить: `getRoles`, `saveDraft`, `loadDraft`, …

Критерий готовности: можно curl'ить эндпоинты, видно ответы.

Критерий готовности: можно собрать команду, добавить услугу,

Критерий готовности: корзина живёт в новой модели, видна из каталога

Действия:
- На проде/деве: `TRUNCATE drafts_table` (через админку HL).

## 10. Открытые вопросы (не блокируют MVP)

Решения по умолчанию проставлены — при реализации можно
перепроверить и поправить дёшево.

1. **Валюта и округление.** Ставка хранится как целое число
   ₽/час. Округление сумм — `round()` до целого (как сейчас).
2. **Лимит ставки.** Без жёстких лимитов; UI принимает любое
   неотрицательное целое.
3. **История изменений ставок.** Не логируем (можно добавить
   позже как audit‑log).
4. **Карточка специалиста.** Минимум: `bitrixUserId`, `rate`,
   `gradeId`. Расширения (комментарий, процент занятости) —
   потенциально в этапе 8.
5. **Один специалист — одна ставка в проекте.** Да, в рамках
   черновика ставка одна. Если нужна разная по ролям —
   добавляется второй специалист с тем же именем (хак), либо
   решается в этапе 8.
6. **Дефолтные часы при добавлении услуги.** Берём из
   `sostav_table.UF_HOURS` (как сейчас).
7. **Глобальный справочник специалистов пользователя.** В MVP
   нет; команда живёт строго в рамках черновика. На этапе 8
   возможен «импорт из предыдущего проекта».
8. **Параллельный расчёт по грейдам.** В Excel оставляем только
   новый формат; старый расчёт убираем (по C2).
9. **Шифрование UF_DATA.** Не делаем в MVP. Защита — на уровне
   прав доступа к черновику (`type` private/shared/public).
10. **Лимит специалистов.** Жёсткого лимита нет. Если упрёмся
    в JSON‑размер `UF_DATA` — введём 50.
11. **Дубли специалистов в команде.** Один `bitrixUserId` —
    одна запись. Попытка добавить второй раз приведёт к
    подсказке «уже в команде, обновить ставку?».
